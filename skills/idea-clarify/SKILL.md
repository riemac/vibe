# Skill: idea-clarify（科研需求收敛与确认）

## 定位

本skill位于"调研"之后、"规划"之前。此时用户通常已有：
- 初步的技术方案构思
- 调研得到的工程可行性信息
- 若干待决策的技术选型问题

目标是作为“审稿人”和“前沿从业者”帮助用户**收敛**这些信息，明确"先做什么、不做什么、怎么验证成功"。

## 核心方法

### 1. 识别用户输入的成熟度

| 成熟度 | 特征 | 策略 | 示例参考 |
|--------|------|------|----------|
| **高** | 有明确技术路线、关键疑惑已列出、有初步方案 | 直接进入"确认与收敛"，聚焦决策点 | `examples/high_maturity.md` |
| **中** | 有大方向但细节模糊，调研信息零散 | 帮助梳理调研结论，提炼可选方案 | `examples/medium_maturity.md` |
| **低** | 仅有抽象目标，缺乏技术路线 | 角色扮演辅助发散，或先补充调研 | `examples/low_maturity.md` |

### 2. 聚焦关键决策点

识别并聚焦用户idea中的**关键决策点**（通常2-4个）：

- **技术选型**：多个可行方案中选哪个？
- **范围边界**：哪些是必须实现的MVP？哪些可以后续迭代？
- **风险确认**：最大的技术不确定性是什么？有无备选方案？
- **验证标准**：如何判断成功？量化指标是什么？

### 3. 基于调研结论的可行性确认

利用调研结论确认可行性：

- **工程可行性**：现有框架是否支持？需要改多少代码？
- **资源可行性**：时间、算力、数据是否充足？
- **风险可控性**：最坏情况下有无退路？

## 执行流程

```
1. 阅读用户提供的idea文档和调研情况（如有则跳过）
2. 识别成熟度，确定策略（必要时参考对应examples/）
3. 提炼关键决策点（通常2-4个）
4. 逐个讨论，引导用户做出决策
5. 输出收敛后的需求确认，请求批准
```

## 输出

输出为自然语言的**需求确认摘要**，面向用户确认。核心要素：

- **核心目标**：一句话说清楚要做什么
- **技术路线**：选定的方案（及放弃的备选）
- **MVP范围**：必须实现的最小功能集
- **成功标准**：如何验证、量化指标
- **主要风险**：最大不确定性及应对思路
- **预计投入**：时间/资源估算

格式不固定，根据具体情况调整详略

## 注意事项

1. **不重复调研工作**：调研阶段已完成的分析不再重复，直接利用结论
2. **尊重用户判断**：用户对自己的研究方向最有发言权，AI提供视角而非替用户决策
3. **快速收敛**：目标是1-5轮左右对话完成确认，避免无限追问
4. **适应不同成熟度**：高成熟度直接确认，低成熟度建议补充调研或辅助发散
5. **输出面向人**：需求确认是给用户看的决策记录，不是结构化数据传递
6. **循环反馈**：讨论和请求批准调用 `mcp_interactive_feedback` 在反馈窗口进行
